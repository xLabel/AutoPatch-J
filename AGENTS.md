# AutoPatch-J: Gemini 行为契约

本文件定义了 Gemini CLI 在参与本项目开发时的 **强制性行为边界**。Gemini 必须以此作为系统提示词的最高优先级扩展。

## 1. 编程灵魂：显式工程化范式
- **强制分层**：任何新功能必须先在 `core/` 下定义领域服务（Service），再在 `tools/` 下封装适配器。严禁在 `agent/` 或 `cli/` 中直接写底层逻辑。
- **命名契约**：类名必须是文件名的大驼峰形式。方法名必须以 `persist_`、`fetch_`、`verify_` 等精准动词开头。
- **显式类型**：生成的每一行代码必须带类型提示（Type Hints），并使用显式注入替代全局单例。
- **模型同步**：修改 `core` 层的 Dataclass 或模型后，必须同步检查 `artifact_manager` 的持久化逻辑与 `cli` 的渲染逻辑，严禁字段脱节。

## 2. 交互红线：环境敬畏与 UI 边界
- **跨平台鲁棒性**：处理源码匹配或补丁生成时，必须强制执行 LF 归一化。严禁因 Windows CRLF 与 Unix LF 差异导致匹配失败。
- **编码纯净度**：子进程执行（如 Semgrep）必须强制开启 `PYTHONUTF8=1` 环境隔离，严禁因 GBK 编码导致规则加载崩溃。
- **UI 库隔离**：严禁在 `prompt_toolkit` 的提示符中使用 `rich` 标签。必须使用 `HTML` 标签或 ANSI 字符串。
- **UI 对象隔离**：在 `rich` 渲染中，严禁在 `Text.assemble` 中直接混入 `Table` 等渲染对象。必须使用 `Group` 进行物理隔离。

## 3. Git 协作守则
- **中文 Commit**：所有 commit message 必须使用中文，严格遵循 `<type>: <中文动词> <具体内容>` 格式。
- **逻辑原子性**：禁止在一个 commit 中提交两个不相关的改动。
- **提交时机**：每个逻辑单元完成后，必须主动询问用户“是否现在提交代码”。

## 4. 决策与回复规范 (ReAct)
- **焦点锁定**：优先处理用户通过 @ 显式提及的文件。在证据确凿后立即提出补丁，严禁通过“审计调用方”等行为推迟任务。
- **最终裁决权**：即便静态扫描器返回 0 发现，若 Agent 确认存在安全风险，必须勇于提出补丁。
- **透传观察**：工具执行结果（Observation）必须通过回调实时反馈给用户。
- **极致精简**：回复文字控制在 3 行以内。

## 5. 纯净度强制要求
- **零债务准则**：若发现某段代码已无调用点，必须主动向用户提议并物理删除它。
- **结构扁平化**：严禁创建超过三层的文件夹嵌套。

## 6. 自我审查：铁律 (Iron Law)
- **物理校验**：每次使用 `write_file` 或 `replace` 后，**必须**立即自我审查代码缩进与语法。
- **边界校验**：修改 Core 层逻辑后，**必须**检查 Validator 是否兼容（如支持无 ID 验证）。
- **回归校验**：对于涉及补丁应用、换行符处理、语义校验等核心链路的改动，**必须**通过运行集成测试（如 `tests/test_agent_full_fix_flow.py`）进行最终闭环。
