# AutoPatch-J: Gemini 行为契约

本文件定义了 Gemini CLI 在参与本项目开发时的 **强制性行为边界**。Gemini 必须以此作为系统提示词的最高优先级扩展，确保其思考模型与项目工程标准高度对齐。

## 1. 编程灵魂：显式工程化范式
Gemini 在生成代码时必须遵循以下“严谨、显式、分层”的工程标准：
- **强制分层**：严格遵守 `core` (领域服务)、`tools` (能力适配)、`agent` (决策调度)、`cli` (交互表现) 的单向依赖结构。
- **职责解耦**：每个模块、类和方法必须职责单一。文件名必须与主类名语义对齐。
- **显式契约**：生成的代码必须包含完整的类型提示（Type Hints）。使用显式的参数注入（Context Passing）替代全局单例或隐式发现。

## 2. 交互红线：环境敬畏
- **严禁擅自修改**：禁止在未得到用户明确指令前，自动修改用户的源码或 `.gitignore`。
- **门禁守则**：实现补丁逻辑时，必须强制落地“物理 -> 语法 -> 语义”三级校验。任何试图绕过校验的简化实现均视为不合格。

## 3. Git 协作守则
- **中文 Commit**：所有 commit message 必须使用中文，严格遵循 `<type>: <中文动词> <具体内容>` 格式。
- **逻辑原子性**：禁止在一个 commit 中提交两个不相关的改动。
- **提交时机**：每个逻辑单元完成后，必须主动询问用户“是否现在提交代码”。

## 4. 决策与回复规范 (ReAct)
- **证据优先**：在研究问题时，严禁基于通用知识猜测。如果没有通过工具看到代码事实，禁止给出任何修改方案。
- **极致精简**：非代码部分的回复文字必须控制在 3 行以内。直接输出思考（Thought）和行动（Action）。
- **实时反馈**：在执行耗时操作前，必须通过信号发出进度说明。

## 5. 跨语言工程化准则
- **通用术语体系**：统一使用业界通用的软件工程术语（如“解耦”、“健壮性”、“显式依赖”），严禁引入带有特定编程语言情怀或偏见色彩的修饰词。
- **评价中立性**：所有的设计决策必须基于客观的工程指标（如维护成本、执行效率）。禁止对任何编程语言的习惯或生态进行主观类比或褒贬。
- **事实驱动**：除了指明扫描目标是 Java 项目这一物理事实外，禁止在讨论中流露任何特定的语言偏好。

## 6. 纯净度强制要求
- **零债务准则**：若发现某段代码已无调用点，必须主动向用户提议并物理删除它。
- **结构扁平化**：严禁创建超过三层的文件夹嵌套，保持项目结构“一眼看懂”。
