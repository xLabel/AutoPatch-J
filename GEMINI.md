# AutoPatch-J: Gemini 协作宪法 (V2.2)

本文件是 AutoPatch-J 项目的核心开发准则，具有最高优先级。Gemini-CLI 在执行任务时必须严格遵守。

## 1. 架构哲学：JAVAer 友好型设计
- **分层原则**：严格遵守 `core` (Service 层)、`agent` (决策层)、`cli` (表现层)、`tools` (适配层) 的单向依赖结构。
- **高内聚低耦合**：每一个类和方法必须职责单一。文件名必须与主类名语义对齐。
- **显式优于隐式**：使用显式的类型提示、单例 context 注入，严禁散落的全局变量或复杂的 IoC 容器。

## 2. 核心命脉：三级校验门禁
任何补丁提案必须通过三层过滤，严禁简化此流程：
1. **物理门禁**：`PatchEngine` 必须确保 `old_string` 在目标文件中唯一匹配。
2. **语法门禁**：基于 Tree-sitter 的 `JavaSyntaxValidator` 必须确认补丁后代码符合 Java 规范。
3. **语义验证**：补丁应用后，`SemanticValidator` 必须自动触发定向重扫，验证原漏洞特征（代码指纹）是否真正消失。

## 3. Agent 行为守则 (ReAct 规范)
- **证据优先**：在未通过 `read_source_code` 看到真实代码前，严禁提交补丁。
- **绝对禁令**：
  - 禁止脑补：只能修复扫描器报告的漏洞，严禁猜测。
  - 禁止伪造：所有 ID 必须来自工具 Observation。
- **极简原则**：生成的补丁必须是“手术级”的，严禁改动无关代码，严禁引入未使用的依赖。
- **自我修正**：若 Observation 返回报错（如语法失败），必须在 ReAct 循环中当场修正，不推卸给用户。

## 4. 交互与 UX 规范
- **非阻塞体验**：补丁门禁必须是异步挂起的（Artifact 模式），不干扰用户后续对话。
- **透明化决策**：必须流式展示 Thought 轨迹，并使用动态 Spinner 展示工具执行进度，降低用户焦虑。
- **上下文精准注入**：基于索引的 `@mention` 必须通过 `CodeFetcher` 切取最相关的语法块（Class/Method），而非全量灌入。

## 5. 开发协作与 Git 规范
- **小步快跑**：一个逻辑点一个 commit。严禁无关重构混入业务提交。
- **Commit 风格**：`<type>: <中文描述>`（无 scope）。
  - `feat`: 新功能；`fix`: 修复；`refactor`: 架构优化；`chore`: 清理。
  - 标题使用中文动词：`新增`、`优化`、`重构`、`移除`。
- **环境敬畏**：严禁擅自修改用户源码和 `.gitignore`。所有副作用必须有确认出口。

## 6. 极致纯净要求
- 保持“小而美”的 MVP 本色。
- 严禁保留没有任何调用的历史入口、调试函数或“备忘”代码。
- 目录结构应保持极致扁平，严禁过度的子文件夹嵌套。
